# `data/` — feature store (mostly gitignored)

This directory is the local feature store. It is **gitignored** (see root
`.gitignore`); the data pipeline (`torontosim.datapipeline`, phase **P01**)
fetches and bakes it from source. Two small, test-critical artifacts are the
exception and **are committed** so the baseline pipeline + tests run without a
network fetch:

| Committed artifact | Why kept | Size |
|---|---|---|
| `data/graph/toronto_drive_graph.json` | Liron's prebuilt OSMnx graph; loaded by `tests/test_graph_mutation.py` + `tests/test_simulation.py`. | ~9 MB |
| `models/demand_model.pkl` | Trained demand model; loaded by `test_simulation.py` (falls back to a heuristic if absent). | ~1.4 MB |

Everything else is **de-committed** and refetched/rebaked by P01:

- `data/graph/toronto_drive_graph.graphml` — 11 MB GraphML; redundant with the
  JSON (JSON is the primary load path, GraphML only a fallback).
- `data/raw/` — raw source pulls: `tmc_raw_data_2020_2029.csv` (TMC turning-
  movement counts, ~28 MB), `data/raw/weather/*` (ECCC climate hourlies), and
  the CKAN export. The legacy weather files with spaces in their names
  (`weather_2023 6_00.csv`) are dropped here; P01 writes clean
  `weather_YYYY_MM.csv`.
- `data/model/{training,validation}_dataset.csv` — engineered training tables;
  rebuilt by `torontosim.model.ingest_real_data`.
- `data/simulation/*` — simulation outputs (e.g. `baseline_result.json`);
  regenerated by the simulator on every run.

## Real-data store (FLO-7, baked by P01 `bake`)

`python -m torontosim.datapipeline fetch && … bake && … verify` (run on the
Spark via `scripts/spark/fetch_and_bake.sh`) produces, all **gitignored**:

- `data/parquet/{centreline,intersections,tmc,signals,bridges,zones}.parquet` —
  normalized from the raw CKAN CSV/GeoJSON pulls (geometry stored as WKT,
  EPSG:4326). Columns + join keys per `research/01` (`centreline_id`/`px`,
  `INTERSECTION_ID`, `ONEWAY_DIR_CODE`).
- `data/catalog.duckdb` — one DuckDB view per parquet for fast local queries
  (`duckdb data/catalog.duckdb "select count(*) from centreline"` ≥ 60k).
- `data/manifest.json` — per-dataset lineage: raw sha256 + license (OGL-Toronto).
- `data/transit/{agency}_{date}.json` — cached real GTFS feeds (TTC/GO/UP) read
  by `transit.gtfs_reader`; the API serves these when present, else the demo set.

These are **opt-in**: the graph defaults to the committed OSMnx JSON
(`TS_GRAPH_SOURCE=centreline` to use the baked Centreline graph), OD defaults to
the gravity baseline (`calibration=ipf_counts` to reconcile against TMC), and
transit falls back to the 509/511 demo set when no cached feed exists.

### Full citywide graph (opt-in, Spark-only, **not committed**)

The committed `data/graph/toronto_drive_graph.json` is the **18k-edge downtown
extent** — it is the test baseline (`test_simulation`, `test_graph_mutation`,
`test_graph_schema`, `test_packaging`) and the offline demo default. Keep it.

The **full citywide** graph (~10× larger, slow to build, heavy to route — a
P11 perf concern) is generated on the Spark and used as a runtime artifact
without replacing the baseline:

```bash
scripts/spark/fetch_and_bake.sh --only centreline,intersections,tmc  # bake first
scripts/spark/build_full_graph.sh                                    # -> data/graph/toronto_full_graph.json
TS_GRAPH_JSON=data/graph/toronto_full_graph.json scripts/run_api.sh  # load it (baseline untouched)
```

To **see and test it from your dev box**, `scripts/spark/serve.sh` starts the
API (8000) + Vite UI (5173) on the Spark in tmux and prints the `ssh -L` line to
forward both to localhost (no ngrok needed):

```bash
TS_GRAPH_SOURCE=centreline scripts/spark/serve.sh    # serves the real Centreline graph
ssh -N -L 5173:localhost:5173 -L 8000:localhost:8000 asus@gx10-4f5f   # then open http://localhost:5173
```

⚠️ **Do not run `python -m torontosim.graph.build_graph --full`** to make the
full graph: it writes to the *fixed* path `toronto_drive_graph.json` and would
**overwrite the committed baseline**. Use `graph.build --source centreline
--out <file>` (what `build_full_graph.sh` does), which writes a separate file.
If the baseline ever gets clobbered, restore it with
`git checkout -- data/graph/toronto_drive_graph.json`.
